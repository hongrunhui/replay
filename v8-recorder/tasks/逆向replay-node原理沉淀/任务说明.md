## 关键结论（速览）

- Replay.io 的核心不是在 V8 Platform API 层做录制，而是在更底层做 **libc/libuv/OS 框架级拦截**，并辅以 **V8 内部插桩与检查点/导航**。
- 仅拦截 Platform API 的时间接口，对 Node.js 程序的非确定性覆盖远远不够: 文件 I/O、网络 I/O、DNS、随机数源、多线程调度都绕过该层。
- replay-node 使用了运行时二进制重写 (trampoline) 进行拦截，并通过一套模板化的“拦截 DSL”(Compose) 统一描述参数/返回值/输出缓冲区与透传策略。

## 可验证的工作流 (用于后续补全/复查)

当需要对不同版本 replay-node 复查或补全时，建议沿用下面的步骤记录证据:

1. 收集二进制元信息: 文件类型、架构、依赖库、导出符号、Build ID 等
2. 识别 record/replay 相关符号簇: `recordreplay::*`、`RecordReplay*`、`node::recordreplay::*`
3. 枚举被拦截的函数集合并分类: libc/POSIX、libuv、平台框架 (macOS CF/dispatch/mach 等)
4. 还原拦截机制: 入口 patch/jmp trampoline、寄存器保存/ABI 翻译、回调拦截
5. 还原事件模型: PassThrough/Disallow/CallbackRegion 的语义与边界
6. 还原线程确定性机制: ordered lock、pointer id、stable hash table 等
7. 还原 V8 集成: 字节码指令、Runtime 函数、检查点与导航相关接口
8. 将结论回写到沉淀文档，并更新与 `v8-recorder` 的差距列表
