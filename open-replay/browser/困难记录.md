## #5 — Chromium 构建时间阻碍迭代

**状态：** 预判
**阶段：** B1' 起
**症状（预期）：** 每次改 patch → `ninja -C out/Default chrome` → 等 10-30 分钟
**根因：** Chromium 4 万+ 文件；改 V8 触发 V8 大半 + 所有依赖它的代码重建
**缓解计划：**
- 用 component build（`is_component_build=true`）—— 重链路径短
- `symbol_level=1` 加快链接
- 不动 chromium 源码，只改我们的 driver dylib —— 是独立 dylib，**重链不到 chromium 主体**，几秒级
- 用 v8_unittests 先调试 V8 改动（小二进制，5 分钟重建）
- 关 NaCl（`enable_nacl=false`）
- 如有 RBE 用 `use_remoteexec`

## #6 — DYLD_INSERT_LIBRARIES 在 hardened runtime 下不传给 Chromium helpers

**状态：** 预判
**阶段：** B5
**症状（预期）：** driver 在主进程加载；子进程（Chromium Helper.app 等）没继承 —— 因为 macOS 在 exec hardened 二进制时清除 `DYLD_*` 环境变量（CS_RUNTIME 标志）
**根因：** macOS 安全特性：以 hardened-runtime 标志启动的二进制，`DYLD_*` 环境变量会被清空
**缓解计划：**
- 构建 Chromium helper bundle 时**不**加 `enable_hardened_runtime`（或等价 GN arg）
- 或：本地用自签证书重签 helpers，去掉 hardened-runtime 标志：
  ```bash
  codesign --force --sign - \
    --preserve-metadata=identifier,entitlements \
    ./Chromium\ Helper.app
  ```
- 或：用 `--entitlements` 加 `com.apple.security.cs.allow-dyld-environment-variables` = true
**参考：** Apple Developer 文档；Chromium 的 `build/config/mac/BUILD.gn` 签名逻辑

## #7 — 多线程回放发散——没有有序锁

**状态：** 已知，B4.3 计划处理
**阶段：** B4 / B7
**症状（预期）：** replay 跑了但 DOM 发散；console 报 `recordreplay::HasDivergedFromRecording` 警告
**根因：** Chromium renderer 是多线程（主线程 + compositor 线程 + worker 线程）。pthread mutex 的 OS 层获取顺序非确定。没有有序锁，回放时线程获取 mutex 顺序可能跟录制时不一样，进而产生不同的执行交错。
**缓解计划：** 在 driver 里实现 `V8RecordReplayCreateOrderedLock`/`Lock`/`Unlock`；driver 维护确定性锁顺序队列
**算法草稿：**
1. 每个注册的 lock 拿到唯一 ID
2. 加锁前：driver 录 (lock_id, thread_id) 事件
3. 回放：driver 阻塞线程直到根据录制顺序轮到它
4. 加锁后：driver 消费事件
**参考：** Replay.io 的 mutex.cc 补丁；`研究/Replay.io-V8补丁清单.md` §12

## #8 — 时钟源绕过（macOS arm64 commpage）

**状态：** 已知（Node.js 经验来的）
**阶段：** B6/B7 大概率
**症状：** 部分时间值没被捕获（clock_gettime 直接返回 commpage 的值，绕过 libc）
**根因：** macOS arm64 commpage 把时间函数 inline 进用户态以避开 syscall 开销
**缓解：** 在 Chromium 层钩 `base::TimeTicks::Now()` 和 `base::Time::Now()`（Blink 用这两个生成时间戳），不靠 libc 拦截
**参考：** open-replay MEMORY.md "Known Issues"

## #10 — V8 snapshot 确定性

**状态：** 预判
**阶段：** B7
**症状（预期）：** replay 从第一条指令就发散，因为 V8 snapshot 状态不一样
**根因：** V8 用二进制 snapshot（`snapshot_blob.bin`）加速启动。如果录制和回放时 snapshot 不一致（编译期嵌入数据不同），后续执行全部发散。
**缓解：** 保证 record 和 replay 用同一份 `snapshot_blob.bin`（同一二进制 = 同一 snapshot）；启动时录 hash
**边界情况：** 如果用户在 record 和 replay 之间升级了 Chromium，全完蛋 —— 必须用同一 Chromium 二进制

## #13 — Driver dylib 必须在链接期就存在（不只是加载期）

**状态：** 已知，已规划绕开
**阶段：** B1' / B2'

**缓解：** Phase B2' = stub driver 作为首个交付物。从同一组 `ForEachV8API` 宏自动生成确保完整。

## #30 — TypeScript 5.0.3 lib.dom.d.ts 引用 Trusted Types 但没声明

**状态：** 已解决
**阶段：** B3 round 12
**症状：** `tsc` 报错：
```
lib.dom.d.ts:7953:25 - error TS2304: Cannot find name 'TrustedScriptURL'
lib.dom.d.ts:9136:32 - error TS2304: Cannot find name 'TrustedHTML'
```
**根因：** TypeScript 5.0.3 自带的 lib.dom.d.ts 用了 Trusted Types API 类型（`HTMLScriptElement.src` 接受 `string|TrustedScriptURL` 等），但这些类型本身需要从 `@types/trusted-types` 加载。chromium tsconfig 设了 `typeRoots: []` 阻止自动发现 @types。
**方案：** 在 lib.dom.d.ts 末尾 append 占位声明：
```ts
declare type TrustedHTML = string;
declare type TrustedScript = string;
declare type TrustedScriptURL = string;
declare interface TrustedTypePolicyFactory {...}
declare var trustedTypes: TrustedTypePolicyFactory | undefined;
```
节点 modules 不在 chromium git 里，所以打成 shell 脚本：`browser/patches/chromium/typescript-trusted-types.sh`。
**日期：** 2026-04-28

## #29 — macOS arm64 SDK 14.2 没有 `__sincosf_stret` 也没公开 `sincosf`

**状态：** 已解决
**阶段：** B3 round 11-12
**症状：** swiftshader `LLVMJIT.cpp:554` 用 `__sincosf_stret`（macOS x86 私有 stret 调用约定），arm64 SDK 不导出。改成 `sincosf` 又发现 arm64 SDK 14.2 只把它当作 inline `__sincosf`，外部链接也找不到。
**根因：** Replay.io chromium 108 时代的 swiftshader 假设 macOS 是 x86。arm64 mac 上 sincosf 既不是 stret 也不是公共符号。
**方案：** 在 LLVMJIT.cpp 里写一个 lambda wrapper：
```cpp
static auto _sincosf_wrapper = +[](float x, float* sp, float* cp) {
  if (sp) *sp = sinf(x);
  if (cp) *cp = cosf(x);
};
functions.try_emplace("sincosf", reinterpret_cast<void *>(...));
```
LLVM JIT 真用到时调它。生产 chromium 走硬件 GPU，根本不进 SwiftShader JIT 路径，所以这个 wrapper 是兜底。
**日期：** 2026-04-28

## #28 — `rollup_plugin.js` 同时用 ESM 和 CJS 语法

**状态：** 已解决
**阶段：** B3 round 9-10
**症状：** rollup 加载 plugin 时报：
- 当 plugin 文件被当 CJS：`SyntaxError: Unexpected token 'export'`（在 `export default function plugin`）
- 当 plugin 文件被当 ESM：`ReferenceError: require is not defined`（在 `const path = require('path')`）
**根因：** `chrome/browser/resources/tools/rollup_plugin.js` 第 8 行 `const path = require('path')`（CJS）+ 第 75 行 `export default function plugin(...)` （ESM）—— 现代 Node 严格区分二者，必须二选一。
**方案：** 改成纯 ESM：
```js
- const path = require('path');
+ import path from 'path';
```
配合 `chrome/browser/resources/tools/package.json` 设 `type: module`，整个目录走 ESM。
**日期：** 2026-04-28

## #27 — Rollup 3.12 + 自动生成的 `rollup.config.js` 在 ESM/CJS 边界上挣扎

**状态：** 已解决
**阶段：** B3 round 8-9
**症状：** `optimize_webui.py` 自动在 `out/Release/gen/.../tmpXXX/rollup.config.js` 写一个 ESM 风格的 rollup 配置（`import plugin from '...'; export default {...}`）。Rollup 3.12 用 Node 的原生 import 机制加载这个配置，但不知道该当 ESM 还是 CJS——根据 package.json 决定。tmpXXX 目录下没 package.json，向上找一层层都没，最终找到我们外层 `chromium/src/package.json` 的 `"type":"commonjs"`（修 #25 时加的）→ 用 require() 加载 → 撞到 `export default` 报 SyntaxError。

## #26 — `~/Documents/code/package.json` 含 `"type":"module"` 污染 chromium 工具脚本

**状态：** 已解决
**阶段：** B3 round 7
**症状：** `tools/polymer/html_minifier.js` 用 CommonJS `require(...)` 但 node 报 `ReferenceError: require is not defined in ES module scope`。错误信息明确指出："This file is being treated as an ES module because it has a '.js' file extension and '/Users/hongrunhui/Documents/code/package.json' contains 'type: module'."

## #25 — Python 3.13 删了 `pipes` 模块

**状态：** 已解决
**阶段：** B3 round 6
**症状：** `build/android/gyp/util/build_utils.py` 走 chrome target 路径（webui 资源处理依赖它）→ `import pipes` → ModuleNotFoundError
**根因：** Python 3.13 移除了 `pipes` 模块。Chromium 老代码用 `pipes.quote(x)`（实际跟 `shlex.quote(x)` 等价，但 chromium 没及时迁移）。
**方案：** 在 `build_utils.py` 顶端做 shim：
```python
import shlex
class _PipesShim:
    quote = staticmethod(shlex.quote)
pipes = _PipesShim()
```
保持 `pipes.quote(x)` 调用不动。
**未修但可能撞：** `build/print_python_deps.py`、`build/env_dump.py`、`build/vs_toolchain.py` 也用 pipes。撞了再修。
**日期：** 2026-04-27

## #23 — Chromium 自带的 `six 1.14.0` 在 Python 3.12+ 上 `from six.moves import` 全炸

**状态：** 已解决（核心 bug，#22 只是表面）
**阶段：** B3
**症状：** 装好 system `six 1.17.0` 后 ninja chrome 还是报 `ModuleNotFoundError: No module named 'six.moves'`。手动 `python3 -c "from six.moves import html_entities"` 在 cwd `tools/grit` 也工作正常 —— 但只要先 `import grit` 就立刻坏。
**根因：**
- `chromium/src/tools/grit/grit/__init__.py` 第 16 行强制把 `chromium/src/third_party/six/src/` 插到 sys.path 头
- chromium bundled six 是 **1.14.0**（2020 年），其 `six.moves` 的 lazy 模块加载机制依赖 Python 3.11- 内部 import 实现，**Python 3.12 改造 importlib 后整套机制失效**
- 所以 `import six` 拿到 1.14.0，`from six.moves import X` 立刻 `ModuleNotFoundError`
**方案：** 把我们 user-site 装的 `six.py 1.17.0` 拷过去覆盖 chromium bundled 的：
```bash
mv chromium/src/third_party/six/src/six.py{,.1.14.0-bak}
cp ~/Library/Python/3.13/lib/python/site-packages/six.py chromium/src/third_party/six/src/six.py
```
**调试过程留档：**
1. system `python3 -c "from six.moves import html_entities"` → OK
2. cwd `tools/grit` 仍 OK
3. cwd `tools/grit` + `sys.path.insert tools/grit` → OK
4. `import grit; from six.moves import ...` → FAIL
5. 看 `grit/__init__.py` —— 它偷偷 `sys.path.insert(0, ...third_party/six/src)`，加载了旧 six
**日期：** 2026-04-27

## #20 — `libbase.dylib` 链接缺 V8RecordReplay\* extern → 全局 dynamic_lookup 是正解

**状态：** 已解决
**阶段：** B1'.5
**症状：** `ninja base` 编完所有 .o 后链接 `libbase.dylib` 失败：~50+ 个 `_V8RecordReplay*` 符号 undefined。
**根因：** `base/record_replay.cc` 把 V8 API 声明为普通 `extern "C"`，靠 `DYLD_INSERT_LIBRARIES` 在运行时由 driver 提供。但 macOS 默认链接器是严格模式，链接期找不到符号就 fail。这就是 OBSTACLES #13 的"必须 link-time 存在"——但实际上 Replay.io 通过 GN 配置把链接器开成 `dynamic_lookup` 模式，把所有未定义符号推迟到 dyld 加载时解析。
**方案：** `chromium/src/build/config/mac/BUILD.gn:49` 把 `-Wl,-undefined,dynamic_lookup` 改成无条件 append（原本 gate 在 `export_libcxxabi_from_executables`）。
```diff
   if (export_libcxxabi_from_executables) {
     ldflags += [ "-Wl,-undefined,dynamic_lookup" ]
   }
+  ldflags += [ "-Wl,-undefined,dynamic_lookup" ]
 }
```
**对 OBSTACLES #13 的修正：** 之前说"driver 必须在 link 期存在" —— 不准确。**driver 必须在 dyld 加载期能找到**。chromium 用 dynamic_lookup 让 link 通过。

## #15 — `gclient sync --force --reset` 触发整个 chromium 重 clone

**状态：** 命中 → 已规避
**阶段：** B1'
**症状：** 我以为 `--force --reset` 是温和地"按 DEPS 重新对齐"，结果它的语义是"删掉子目录从头 clone"。命令开始后 5 分钟还在 `git clone --no-checkout` 阶段（remote 公布要传 62GB），按 ~10MB/s 算要 100+ 分钟，且这才是 chromium/src，下面还有几十个 third_party 子模块。
**根因：** `--force` 让 gclient 把"修改过的"被认为冲突的 git 仓库整个重新拉。`--reset` 让 gclient 重置子模块到 DEPS 里指定的 commit。两个一起用就是核武器。我们的 chromium/src 几年没动 + 参数过激，gclient 选择了"全删重来"。
**试过：** `pkill -9 -f gclient.py sync` + `pkill -9 -f deltaBaseCacheLimit`（git clone 子进程）+ 删除 `/Users/hongrunhui/Documents/code/_gclient_src_t4soadv9` 临时目录。
**方案：**
- 不用 `--force --reset`，先跑朴素 `gclient sync` 看缺什么
- 或者**先跳过 sync**，直接 `gn gen` 试试，根据错误信息判断哪些 deps 缺
- 如果只是某个子模块旧/缺，单独 `gclient sync --revision src/v8@xxx` 这种精准操作
**参考：** `/tmp/gclient_sync.log` 当时输出
