## Phase B2' — Stub Driver

**目标：** 一个 dylib，导出全部 113 个 V8/Skia/Blink driver extern 符号，全是 no-op 返回默认值。让 chromium 能链接通过。
**工期：** 2-3 天。
**替代：** 旧 Phase B2（port V8 补丁）—— 不再需要；Replay.io 已做。

### B2'.1 — Stub 生成器脚本
- 解析 `chromium/src/base/record_replay.cc` 里的 `ForEachV8API` 和 `ForEachV8APIVoid` 宏
- 自动生成 `browser/driver/src/stubs.cc`
- 每个函数：`extern "C" ReturnType Name Formals { return DefaultValue; }`
- void 变体：空函数体
- Skia 符号（~19）和 Blink 回调（~10）—— 从 `研究/Driver-API契约.md` 手工列出来

### B2'.2 — Driver 构建
- 新建 `browser/driver/CMakeLists.txt` 或 `build.sh`（参考现有 `open-replay/driver/build.sh`）
- 输出：`browser/driver/build/libopenreplay_browser.dylib`
- 全部符号导出（macOS 默认可见性 —— **不**用 `-fvisibility=hidden`）
- 验证：`nm -gU build/libopenreplay_browser.dylib | grep -c V8RecordReplay` ≈ 59

### B2'.3 — 把 driver 接进 chromium 构建
- 选项 A：在 chromium 的 `chrome/BUILD.gn` 里加运行时依赖
- 选项 B：把 dylib 放在 chromium 的 `out/Default/`，用 DYLD_INSERT_LIBRARIES 在运行时加载

  需要看 Replay.io 自己的 `chrome/BUILD.gn` 怎么处理。

  可能：chromium 链接期望系统装有 `libopenreplay.dylib` 之类。

  **行动：** 在 chromium 里搜 BUILD.gn 是否引用 "libopenreplay" 或 "Replay" 链接。定位放置目标。

### B2'.4 — 第一次 chromium 链接
- `autoninja -C out/Default chrome`
- 期望：chromium 二进制产生。可能不能跑（init 阶段碰到 stub 返回会崩）

**B2' 成功标准：** `out/Default/Chromium.app` 存在（端到端构建通过）。

## Phase B4 — Driver：真实的录制骨架

**目标：** chromium 跑的时候 driver 写出 `.orec` 文件。文件含头 + 几个真实事件。
**工期：** 1 周。

### B4.1 — 进程类型识别
- 读 chromium 命令行的 `--type=` 参数
- 区分：browser / renderer / gpu / utility / network
- 各自写 `<role>-<pid>.orec`

### B4.2 — 实现基础状态函数
- `V8IsRecording`、`V8IsReplaying`、`V8GetRecordingId` —— 读环境变量
- `V8RecordReplayValue`、`V8RecordReplayBytes` —— 写 .orec
- `V8RecordReplayPrintVA` —— 写日志文件

### B4.3 — 实现事件录制器
- `V8RecordReplayOnAnnotation` —— 存事件
- `V8RecordReplayOnNetworkRequest`、`OnNetworkStreamData`、`OnNetworkStreamEnd` —— 存
- `V8RecordReplayBrowserEvent` —— 存

### B4.4 — 实现 disallow / passthrough 作用域
- 维护每线程深度计数器
- `V8RecordReplayBeginDisallowEvents` / `End...`
- `V8RecordReplayBeginPassThroughEvents` / `End...`

### B4.5 — 验证 .orec 输出
- 启动 chromium，访问 `https://example.com`，退出
- 检查每进程的 `.orec`：都该有头 + 合理数量的事件
- 简单页面加载总字节量在 1-50MB 之间

**成功：** 改造后的 chromium 给所有进程产出格式正确的 `.orec` 文件。

## Phase B6 — 实现录制端确定性层

**目标：** driver 录下所有非确定性，让 replay 成为可能。
**工期：** 1 周。

- 时间：实现 `V8RecordReplayValue` 给时间/日期函数（复用 Node.js driver 的 libc 拦截逻辑）
- 随机：`Math.random` 已经通过 V8 补丁录制 —— 验证 `V8RecordReplayBytes("MathRandom")` 被捕获
- 文件 I/O：从 Node.js driver port 现有的 fs.cc 拦截
- 网络：`V8RecordReplayOnNetworkRequest` 等 —— 存请求体和响应
- 输入事件：`V8RecordReplayOnMouseEvent`、`OnKeyEvent` —— 存
- 有序锁：`V8RecordReplayCreateOrderedLock`、`Lock`、`Unlock` —— 实现确定性队列（见 `困难记录.md` #7）

**成功：** 录制捕获了确定性回放所需的一切（事件计数 + 抽样内容验证）。

## Phase B7 — 第一次 replay（关键集成测试）

**目标：** replay 复现同一次页面加载。

### B7.2 — 一次 record→replay 往返
- 录 `file:///tmp/static.html`（最简单的目标）
- replay；通过 CDP 截 DOM
- 跟录制时的 DOM diff；应当完全一致

### B7.3 — 迭代解决发散
- 用 AssertValue 字节码（Replay.io 的 V8 已经发了）来定位
- 每个新发散 → 找源头 → 加拦截/录制 → 重录 → 重放

**成功：** 至少一个页面加载能 bit-identical 回放。
