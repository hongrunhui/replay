# Open Replay — 浏览器运行时回放架构

> 状态：**草案 v0.2** — Phase B0 完成（研究），Phase B1' 进行中
> 版本：0.2（2026-04-27，根据"chromium fork 已在本地"做了重大调整）
> 目标基底：Replay.io 开源 chromium fork（vanilla Chromium 108 + V8 11.7）

---

## 一、产品目标

把网页运行时录一次，之后能确定性回放并附带完整的时间旅行调试。能力对齐当前 Node.js 流水线，但作用域是浏览器：

- 用户启动我们改造过的 Chromium，浏览/点击/滚动 → 我们录
- 之后用 replay 模式重启浏览器 → DOM、网络、绘制完全一致
- DevTools 接入 replay → 任意行打断点、前进/后退单步、任意执行点查任何变量

**这不是 session replay（rrweb 那种 DOM diff 回放）**。这是真正的 runtime 自身在重放，靠的是录制下来的输入数据。每一次 `Math.random()` 返回相同的数。每一次 `fetch()` 拿到相同响应。每一次 `requestAnimationFrame` 在相同的执行点触发。

## 二、整体架构（与 Node.js 流水线原理一致）

```
┌──────────────────────────────────────────────────────────────┐
│        Replay.io Chromium Fork（已在本地，BSD 协议）         │
│                                                              │
│  Browser Process  ──┐                                        │
│  Renderer Process ──┼── 都通过 DYLD_INSERT_LIBRARIES         │
│  GPU Process      ──┤   加载 libopenreplay_browser.dylib     │
│  Network Service  ──┤                                        │
│  Utility Procs    ──┘                                        │
│                     │                                        │
│          ┌──────────▼────────────┐                           │
│          │  V8 11.7（已打补丁）   │  ← 7 个字节码           │
│          │  - 进度计数器          │    8 个 runtime 函数    │
│          │  - 插桩点              │    脚本 ID 录制         │
│          │  - 断言值              │    Math.random 录制     │
│          └────────────┬───────────┘                          │
│                                                              │
│  Blink/Skia/net/ 也全部已打补丁（Replay.io 完成）           │
│  调用约 113 个 extern "C" 符号到 driver                      │
└───────────────────────┼──────────────────────────────────────┘
                        │
                        ▼
           ┌────────────────────────────┐
           │  libopenreplay_browser.dylib│
           │  （我们要写的部分）         │
           │                            │
           │  - 实现 113 个 extern 符号 │
           │  - 进程类型识别            │
           │  - 有序锁服务器            │
           │  - .orec 写入器/读取器     │
           │  - 检查点 fork()           │
           └────────────┬───────────────┘
                        │
                        ▼
             ~/.openreplay/recordings/
                   <uuid>/
                      ├── browser.orec   ← 每进程一个 .orec
                      ├── renderer0.orec
                      ├── gpu.orec
                      └── net.orec
```

**核心洞察**：Replay.io 的所有 chromium 端补丁都已经做好（V8 字节码、Blink 输入事件、Skia 绘制录制、net/ 钩子）。我们的工作收敛到**实现一个开源版 driver，去满足他们这个 fork 期望的 extern API 合同**。这比"自己 fork 个 chromium 改 70 个文件"轻松得多。

## 四、Chromium 进程模型——我们怎么处理

Chromium 拆成多个进程。对录制/回放来说，全部平等对待：

| 进程 | 录什么 | 怎么注入 |
|------|--------|----------|
| Browser（主） | UI 状态、窗口管理、Tab 创建 | 启动时 `DYLD_INSERT_LIBRARIES` |
| Renderer | **JS 执行、DOM、fetch、Blink 事件**（主要目标） | 从父进程继承 preload |
| GPU | Skia 光栅化输入、Compositor 帧输入 | 继承 |
| Network service | HTTP 请求/响应、DNS | 继承 |
| Utility | （多数忽略） | 继承 |

**跨进程协调**：
- 每进程写自己的 `<role>-<pid>.orec` —— 不需要共享写锁
- Mojo IPC **不录** —— 跨进程的确定性靠"两边各自确定性回放 + 录制好的输入"自然涌现（Replay.io 的核心洞察）
- **有序锁**（`V8RecordReplayCreateOrderedLock`）用来串行化无法独立录制的共享状态访问（比如 browser 和 renderer 的共享内存区）

**沙箱处理**（见 `困难记录.md` #1）：
- macOS renderer 沙箱会拦截 DYLD_INSERT_LIBRARIES
- **MVP 计划**：用 `--no-sandbox` 启动。Phase B7 跑通端到端后再回头解决
- 长期：签名 dylib + 修改 sandbox profile

## 五、非确定性来源（浏览器 vs Node.js）

Node.js driver 已经处理的（直接 port）：
- `gettimeofday`、`mach_absolute_time`、`clock_gettime` → 时间拦截
- `read`、`write`、`open`、`close`、`stat` → 文件系统拦截
- `recv`、`send`、`connect` → socket 拦截
- V8 里的 `Math.random()` → RNG cache 录制

**仅浏览器**有的非确定性（新工作）：

| 来源 | 入口 | 拦截方案 |
|------|------|----------|
| **鼠标/键盘输入** | Chromium 输入管线 → Blink InputEventManager | Replay.io 已经在 Blink 里发 `V8RecordReplayOnMouseEvent`/`OnKeyEvent`；driver 实现这些回调 |
| **requestAnimationFrame 时序** | Compositor → Renderer via IPC | 在 `ScriptedAnimationController::ServiceScriptedAnimations` 录帧时间戳（Replay.io 已做） |
| **DOM 事件顺序** | Blink 事件队列 | 构造性确定（事件按入队顺序派发）；只要录好输入事件即可 |
| **IndexedDB / localStorage** | Blink → 磁盘 | 已被 fs 拦截覆盖（底层是 LevelDB） |
| **WebSocket / SSE** | Network service socket | 已被 socket 拦截覆盖 |
| **Web Crypto API** | Blink → BoringSSL | 跟 Node.js 一样：钩 BoringSSL 的 `RAND_bytes_ex` |
| **performance.now()** | Blink → base::TimeTicks | 钩 `base::TimeTicks::Now()` |
| **Scheduler / 任务顺序** | Blink scheduler | 棘手——主线程任务队列。可能要靠有序锁 |
| **Canvas/WebGL** | Skia | 绘制录制（Replay.io 已加 19 个 SkRecordReplay 符号） |
| **音频（WebAudio）** | media/ | **MVP 不支持**（Replay.io 也没做） |
| **WebRTC** | media/ | **MVP 不支持** |

## 六、.orec 格式——浏览器版扩展

现有 `.orec` 格式（64B 头 + 事件流 + 检查点索引 + 32B 尾）原样可用，但加几种新事件：

| 类型 | 名称 | 范围 |
|------|------|------|
| `0x01` | VALUE | 已有 |
| `0x02` | BYTES | 已有 |
| `0x20` | METADATA | 已有 |
| `0x30` | PAINT | 新——skia 命令缓冲 |
| `0x31` | INPUT_EVENT | 新——鼠标/键盘 JSON |
| `0x32` | RAF_TIMESTAMP | 新——动画帧时序 |
| `0x33` | OBJECT_ID_BINDING | 新——给 TrackObjectId 字节码用 |
| `0x40` | NETWORK_REQUEST | 新——HTTP 请求 |
| `0x41` | NETWORK_RESPONSE | 新——HTTP 响应 |

每个 chromium 进程一份 `.orec`，全部放在 `~/.openreplay/recordings/<uuid>/` 下。回放引擎跨文件协调读取。

## 八、主要风险（在 `困难记录.md` 里追踪）

1. **沙箱拦截 DYLD_INSERT_LIBRARIES 进 renderer** — 缓解：MVP 用 `--no-sandbox`
2. **V8 启动期 driver 还没 init 好就被调用** — driver 必须在 `__attribute__((constructor))` 里完成所有初始化
3. **chromium 链接期需要全部 113 个符号存在** — 必须先有 stub driver
4. **macOS arm64 commpage 绕过 DYLD_INTERPOSE** — 跟 Node.js 一样的问题
5. **多线程回放发散——必须实现有序锁** — Node.js 跳过了，浏览器跳不过
6. **Skia 绘制录制是大工程**（19 个符号）— B9 处理
7. **chromium 108 是 2 年前的版本** — demo 时浏览器看着旧；接受

## 九、首个里程碑成功标准

**最小可用浏览器回放（Phase B7 末）：**

1. Replay.io chromium fork + 我们的 driver 在本地构建成功
2. 启动 `chromium-replay record https://example.com` —— 录制静态页面加载
3. 启动 `chromium-replay replay <uuid>` —— 重启 chromium，确定性回放同一次加载
4. 网络事件匹配（同样的字节、同样的顺序）
5. 最终 DOM 状态匹配（diff < 10 个节点）

**首期不做**：时间旅行、交互式回放、DevTools 连接、带 JS 动画的动态页面。
