## 2026-04-28（晚上）— Phase B6.3 半实现：有序锁录序列 + Disallow scope 录制

### 实现
- `V8RecordReplayCreateOrderedLock(name)` 改返回 `FNV-1a(name)` 作为 lock id（同名在 record/replay 两次执行产生同一 id，无需 driver 维护映射）
- `V8RecordReplayOrderedLock(id)` / `OrderedUnlock(id)` 录 (lock_id, thread_id) 到 BYTES 事件
- `V8RecordReplayBeginDisallowEvents` / `BeginDisallowEventsWithLabel` / `EndDisallowEvents` / `BeginPassThroughEvents` / `EndPassThroughEvents` 也录上（这些是 `[RUN-1039]` log 的对应函数，chromium 高频调用）

### 重要限制
**这是 record-only 半实现**。回放模式仅推进游标，**不真正阻塞线程**。完整实现要求 replay 时让线程等到自己的轮次（按 record 录的 (lock, thread) 顺序）——需要 pthread_cond_t + per-thread waiters，下次再做。

### 验证
example.com headless 12s：browser events 566 → 588 (+22)，10096 字节。replay 应该一样能加载（结构没变，只多了几种 why_hash）。

### 真正的下次入口
**Phase B6.4：libc 拦截层 port from Node.js driver**。这是 deterministic replay 的最后一公里——时间/随机/fs/socket 都要在 driver 这层拦下来。Node.js driver 的 `intercept/{time,random,fs,net}.cc` 直接复用，DYLD_INTERPOSE 机制对 chromium 同样有效。

之后 **Phase B6.3 完整版**：让 OrderedLock 在 replay 时真阻塞线程。

## 2026-04-28（傍晚）— Phase B6.1 + B6.2：11 个 On* 遥测函数真实化

### 一句话总结
把 11 个先前 weak no-op 的 On* 系列函数（5 网络 + 4 输入 + 2 浏览器/注解）改写为真实序列化记录，单次 example.com headless 加载事件数从 466 → 566（+21%），4 进程 .orec 全部增长。

### 实现要点
record_real.cc 加 3 个静态辅助函数：
- `AppendString(buf, s)` —— 4 字节长度前缀 + 字节内容
- `AppendNum<T>(buf, v)` —— 原始字节
- `RecordOrConsume(why, buf)` —— record 模式 WriteBytes，replay 模式只推进游标（暂不验证内容；后续 AssertValue 阶段补）

state.h/cc 加：
- `DriverState::ReplayConsumeEvent(why)` —— 调 `reader_->NextEvent(why)` 推进游标
- `DriverState::recorded_events()` —— 暴露 writer 的 event_count

11 个真实实现：
| 函数 | 参数序列化 |
|---|---|
| OnNetworkRequest | id + kind + bookmark |
| OnNetworkRequestEvent | id |
| OnNetworkStreamStart | id + kind + parentId |
| OnNetworkStreamData | id + offset + length + bookmark |
| OnNetworkStreamEnd | id + length |
| OnEvent | event + before |
| OnMouseEvent | kind + clientX + clientY + synthetic |
| OnKeyEvent | kind + key + synthetic |
| OnNavigationEvent | kind + url |
| OnAnnotation | kind + contents |
| BrowserEvent | name + payload |

### 验证

Record example.com headless 12s：
```

gpu:       3039 → 3396 字节
network:   3294 → 2648 字节  ← 减少（可能因为 streaming buffering 顺序变化）
renderer:  1816 → 2500 字节
```

Replay 同一 session：
```
browser:   566 events loaded
gpu:       196 events loaded
network:   152 events loaded
renderer:  143 events loaded
```

Chromium 不崩、reader 全部成功打开、events 数跟 record 完全一致。

### 限制 / 已知

- replay 模式**只推进游标**，没验证事件内容是否匹配。如果 record/replay 时序漂移，driver 不会报错。后续做 AssertValue 时再加 hash 校验。
- 未实现的关键 On* 函数：`V8RecordReplayCurrentReturnValue`、`V8RecordReplayNewBookmark`（这俩有返回值，需要更深的 record/replay 配对逻辑），任务 #18 有序锁也还是 stub。

### 下次会话入口

继续 **Phase B6.3：有序锁** （task #18）。多线程 V8 必须的——renderer 进程有 10+ 线程，pthread_mutex 获取顺序非确定，没有有序锁回放会发散。

之后是 libc 层时间/随机/fs/socket 拦截（从 Node.js driver port），那是真"deterministic replay"的最后一公里。

## 2026-04-28（下午）— Phase B7：第一次完整 record→replay 往返跑通

### 一句话总结
本机 Replay-Chromium **headless** 加载 `https://example.com` 12 秒，driver 录下 4 个进程的 .orec 共约 16KB（466+175+190+103 = 934 events）；切换到 replay 模式重启同一 chromium，driver 全部 4 个进程**成功打开 reader 并消费录制事件**，chromium 不崩。**这是 Phase B7 首个完整往返。**

### 触发的修补

1. **B7.2: atexit() 让 .orec 干净 close**
   - 进程析构时 RecordingWriter 写 32B tail（含 0xDEADBEEF sentinel）。但 chromium 子进程（renderer/gpu/network）经常是 SIGKILL 退出，不跑 C++ 静态析构。在 DriverState 构造里 `std::atexit(...)` 注册一个 lambda 关 writer，让 normal-exit 进程能正确关。
   - 验证：`xxd browser-*.orec | tail -3` 看到尾部 `efbe adde` = 0xDEADBEEF 正确。
   - 限制：SIGKILL 仍跳过 atexit。helper 进程的 .orec 通常仍残尾。需要 #2 兜底。

2. **B7.3 引发的修补 1: replay 模式按 role 找 .orec**
   - 录制时 .orec 命名 `<role>-<pid>.orec`，pid 是录制时进程的。回放时新进程 pid 不一样，找不到原文件。
   - 修法：录制时按 pid 命名（避免冲突），回放时 scan 目录找 `<role>-*.orec` 字典序最早一份匹配。
   - 文件：`state.cc::ResolveSessionAndOpen()` 加分支。

3. **B7.3 引发的修补 2: reader 容忍坏 tail**
   - 部分 helper .orec 因 SIGKILL 缺尾，reader 严格校验 0xDEADBEEF 拒绝加载。
   - 修法：`recording.cc::Open()` 把 ParseTail 失败从 fatal 降为 WARN，把整个 header 之后内容当事件流（跳过 checkpoint index）。
   - 验证：4 个进程全部 load 成功，包括 3 个 tail 损坏的。

### 数据观察

录制 example.com headless 12 秒：
```
browser-18075.orec   8022 字节   466 events  (tail 完整：atexit 跑了)
gpu-18096.orec       3039 字节   175 events  (tail 损坏：SIGKILL)
network-18099.orec   3294 字节   190 events  (tail 损坏：SIGKILL)
renderer-18102.orec  1816 字节   103 events  (tail 损坏：SIGKILL)
```

事件数比预期少很多。原因：我们绝大多数 `V8RecordReplayOn*` API（network/mouse/key/navigation）都是 **weak stub no-op**，chromium 调它们时数据被丢。只有少数路径（被 record_real.cc 真实实现的 `V8RecordReplayValue` `V8RecordReplayBytes` `V8RecordReplayAddMetadata`）真录到 .orec。

### Replay 模式行为

Chromium 在 replay 模式下：
- ✅ `V8IsReplaying()` 正确返回 true（driver 真实实现）
- ✅ 启动、加载 about:blank、12 秒后被 SIGINT 干净退出
- ✅ 4 进程都打开 reader，准备好"喂数据"
- ⚠️ 但 chromium 可能还是真的跑了 ——我们没观察到 chromium "拒绝走真实路径"的证据，因为 driver 没记到关键的 OnNetworkRequest 等数据，chromium 看驱动也"没数据"，可能就走了 fallback 真实路径

### 下次会话入口

**Phase B6 优先级排序**（按发散影响排）：
1. **B6.1** `V8RecordReplayOnNetworkRequest/StreamData/StreamEnd/RequestEvent`：网络是最大确定性源，必须真录真回放
2. **B6.2** `V8RecordReplayOnMouseEvent/OnKeyEvent/OnNavigationEvent`：用户输入回放，做 Phase B8 的基础
3. **B6.3** `V8RecordReplayCreateOrderedLock/Lock/Unlock`：多线程 V8 必须，否则 renderer 多线程会发散
4. **B6.4** 时间/随机：libc 拦截层（移植 Node.js driver 的 time.cc/random.cc）

每完成一个就重跑 B7 record→replay，看事件数变化和发散情况。

### 顺手发现

- `lld 16` 自带的 chromium 工具链在 macOS 15.6 + SDK 14.2 上 **segfault** 链接 mach-o。
  workaround：`use_lld = false`，改用 Apple 系统 ld。
- `record_replay_inline.h` 是 chromium fork 留的"build dependency ordering"应急 fallback，但**只实现了 `RecordReplayAutoPassThroughEvents` 一个 RAII**——`AutoDisallowEvents` 没有，导致 partition_alloc 撞墙。我添加了一份 inline 版本（`recordreplay::AutoDisallowEvents` 通过 `dlsym(RTLD_DEFAULT)` 在运行时找 driver 符号）但发现 partition_alloc 实际包含的是 `record_replay.h` 而不是 `_inline.h`，所以最后采用了"注释掉调用"的简单路径。inline 版的代码留着，未来需要时直接用。

### 关键认知

**Replay.io 在 macOS 上 chromium ↔ driver 的链接模型 = 全局 dynamic_lookup**。

我们之前担心"driver 必须在链接期就存在"（OBSTACLES #13）—— 部分对：**符号必须在运行期能解析**。但 chromium 的 macOS link 配置允许通过 `-Wl,-undefined,dynamic_lookup` 把所有 V8RecordReplay\* 等 extern 推迟到 dyld 加载时再解析。这就是 Replay.io 闭源 driver 的工作方式。

我们的开源 stub driver（`libopenreplay_browser.dylib`）正是要做这件事——通过 `DYLD_INSERT_LIBRARIES` 在 chromium 启动时被 dyld 加载到所有进程，提供这 99 个符号。

OBSTACLES #13 已根据这个新理解更新。

## 2026-04-23 — Phase B0：起步与研究

### 本次会话目标
搞清楚做 Replay.io 级别的浏览器 replay 实际涉及什么。用户明确要"replay.io 级别"，不是 session-replay。用户接受了 Chromium fork 的承诺。

1. 盘点 `replay-github/chromium-v8/`（来自 Replay.io Chromium 108 fork 的 V8）—— 完整 V8 补丁清单。**产物：** `研究/Replay.io-V8补丁清单.md`
2. 用公开博客 + 他们的 GitHub fork 调研 Replay.io 的 Chromium 端架构。**产物：** `研究/Replay.io-Chromium架构调研.md`
3. 起草 `架构.md`、`路线图.md` —— 11 阶段，6-10 周估计。

### 关键发现

**发现 #1 — Replay.io 的 Chromium 改动很少。** 他们的主要工作是 V8 补丁 + 一个胖 driver（libopenreplay）。Chromium 端的改动只有十几个文件：startup 时 driver attach、Blink 输入事件钩子、Skia 绘制录制、net/ 小改、白名单。**没改 Mojo、没改沙箱、没改 cc/。** 这是可做的。

**发现 #2 — 我们已有的工作能复用 70%。** Node.js v20（V8 11.3）的 V8 补丁直接 apply 到 Chromium 116（V8 11.6）几乎没事。只缺 3 个字节码 + 3 个 runtime 函数（vs Replay.io 的 7 个）。`.orec` 格式、`raw_syscall.h`、InterceptGuard 全部不动 port。

**发现 #3 — Maglev 重要。** Node.js v20 默认不开 Maglev，所以我们跳过了。Chromium 117+ 默认开 Maglev。要么给 Maglev 的 graph builder 打补丁，要么 `--no-maglev` 启动。**决定：MVP 关 Maglev**（更简单），后续阶段加补丁。

**发现 #4 — 沙箱是大未知。** Replay.io 公开文档没解释怎么把 dylib 加载进 renderer（renderer 是被沙箱保护的）。最佳猜测：他们整个禁用沙箱。**我们的计划：MVP 用 `--no-sandbox`。** 如果 macOS 上 chromium helpers 还是因为 hardened runtime / 代码签名挡住 DYLD_INSERT_LIBRARIES，我们可能要重签名或换注入机制。

**发现 #5 — 有序锁不能不做。** Replay.io 的 `V8RecordReplayAddOrderedPthreadMutex` 回调把每个 V8 内部的 pthread_mutex 都包了一层。没这个，多线程 V8 回放会发散。Node.js 因为我们跑单线程所以没事。Chromium renderer 有 10+ 线程；我们 **必须** 在 Phase B7 之前在 driver 里实现有序锁。

### 已记录的决策

| # | 决策 | 理由 |
|---|------|------|
| 1 | 目标 Chromium 116 | V8 11.6 接近我们 Node v20 的 V8 11.3；够新 |
| 2 | 每进程 .orec 文件 | 避免跨进程写锁；更简单 |
| 3 | MVP 用 `--no-sandbox` | 不被沙箱问题阻塞；调试工具不用上线 |
| 4 | MVP 用 `--no-maglev` | 先跳过 Maglev 补丁；后续加 |
| 5 | 保留现有 .orec 格式 | 加新事件类型（PAINT、INPUT_EVENT、RAF）；不破格式 |
| 6 | 所有进程都禁沙箱 | DYLD_INSERT_LIBRARIES 进 renderer 的前提 |
| 7 | 复用现有 CLI、server、DevTools 包 | 加浏览器特有命令，不重写 |
